# 游戏当前规则说明

本文按当前代码实现整理，作为后续玩法、美术和前端迭代的规则基线。

## 一、世界

- 默认种族：`human`、`elf`、`orc`。
- 默认资源：`food`、`wood`、`stone`。
- 默认地形：`plain`、`forest`、`hill`、`water`、`mountain`。
- 默认天气：`clear`、`rain`、`drought`、`storm`。
- 默认地图尺寸是 `32x20`，Web 入口也默认创建 `32x20` 世界。
- 地形随机生成：水约 8%，山约 8%，森林约 22%，丘陵约 17%，其余平原。
- 水和山不可通行，不能被普通扩张命令占领。
- 每族初始有 1 个出生地。出生地会尽量拉开距离，大图优先曼哈顿距离 8，空间不足时逐级降低。
- 每族初始资源：`food=120`、`wood=80`、`stone=40`。
- 每族出生地初始：15 人口、5 士兵、2 房屋，其中 10 农民、5 闲置人口。
- 初始外交关系为 `neutral`。

## 二、Tick 与战略回合

- 每次 tick 会让世界时间 `+1`。
- 世界暂停后，tick 不再推进。
- 每 tick 顺序：天气持续时间递减、NPC 被动结算、领土归属修正、发现新种族。
- 默认每 5 tick 触发一次战略回合。
- 普通 tick 不调用 LLM。
- 战略回合会并发请求所有未淘汰首领决策。
- 某个首领计划通过校验后，会立刻更新该阵营的 `last_plan_snapshot`，前端可以先展示，不必等所有首领都完成。
- 所有可行动首领都提交合法计划后，再按种族 ID 排序执行 NPC 主动命令。
- 每 tick 结束会写入事件：`Tick N completed`。

## 三、地块、人口与容量

- 地块容量由地形和房屋决定：`capacity = terrain_base + houses * 5`。
- 基础容量：
  - 平原 `60`
  - 森林 `45`
  - 丘陵 `35`
  - 水 `0`
  - 山 `0`
- 人口职业：`idle`、`farmer`、`lumberjack`、`miner`、`builder`。
- 人口自然增长每 tick 检查一次。
- 食物低于 20 时不增长。
- 干旱和风暴地块不增长。
- 每个安全、有人、未满容量的己方地块每 tick 最多增长 1 个 `idle`。
- 每增长 1 人消耗 2 食物，并保留 20 食物底线。
- 训练士兵会消耗闲置人口；士兵不是凭空生成。
- 如果一个己方地块既没有本族人口也没有本族士兵，会失去归属。

## 四、资源

- 每 tick 人口和士兵会消耗食物：`max(1, (population + soldiers) // 30)`。
- 食物不足时，食物清零，并按缺口造成饥荒人口损失：`max(1, shortage // 2)`。
- 农民产粮：
  - 平原除数 `3`
  - 森林除数 `6`
  - 丘陵除数 `10`
  - 水和山不产粮
- 伐木工产木：森林除数 `5`，其他可通行地形除数 `10`，风暴时除数额外 `+3`。
- 矿工产石：丘陵除数 `5`，其他可通行地形除数 `14`，风暴时除数额外 `+4`。
- 天气倍率：
  - `clear`：`1.0`
  - `rain`：`1.35`
  - `drought`：`0.35`
  - `storm`：`0.45`
- 人口任务可额外产资源：
  - `farm`：`max(1, workers // 3)` 食物
  - `gather_wood`：`max(1, workers // 4)` 木材
  - `mine_stone`：`max(1, workers // 5)` 石料
- `build` 会把闲置人口转为 builder，并自动消耗木材建房；每座房屋消耗 10 木材。
- `resource_orders` 中 `spend`、`trade`、`tribute` 会扣本族资源；当前还不会把资源转移给对方。
- `reserve` 可通过校验，但当前没有执行效果。

## 五、天气

- 上帝改天气时会设置持续时间：
  - `rain` 默认 5 tick
  - `drought` 默认 5 tick
  - `storm` 默认 3 tick
  - `clear` 持续时间为 0
- 每 tick 天气持续时间递减，到 0 后自动变回 `clear`。
- 风暴地块每 tick 会造成 1 人口损失。
- 风暴在偶数 tick 若该格有士兵，会造成 1 士兵损失。
- 干旱每 3 tick 造成一次人口损失：`max(1, population // 30)`。

## 六、人口命令

- `population_orders` 支持：`farm`、`gather_wood`、`mine_stone`、`build`、`settle`、`train`、`defend`、`attack`。
- `farm`、`gather_wood`、`mine_stone`、`build` 只能把目标地块的 `idle` 转成对应职业。
- `workers` 表示本回合新投入人数，不是最终职业总人数。
- 职业任务目标必须是本回合开始时已经拥有的地块，不能在同一计划里先扩张再安排新地块工作。
- `train` / `defend` 会训练士兵，数量为 `max(1, workers // 5)`，且受目标地块闲置人口和人口三分之一上限限制。
- `attack` 作为人口任务目前没有执行效果。
- `settle` 作为人口任务会尝试执行一次默认迁民，但更推荐使用 `territory_orders` 的 `claim` / `settle`。

## 七、领土与迁民

- 首领只能操作可见坐标。
- `claim` 只能用于无主地块。
- `settle` 可用于无主地块或本族已拥有地块。
- `claim` / `settle` 目标必须在地图内、可通行、非敌方、与本族领土相邻。
- `settle` 可指定：
  - `origin`：迁民来源地块，必须与目标相邻。
  - `profession`：迁移职业，可为 `idle`、`farmer`、`lumberjack`、`miner`、`builder`。
  - `amount`：迁移平民人数。
- `claim` / `settle` 未指定 `amount` 时，使用保守默认值 `2`。
- 迁民只移动非士兵人口。
- 指定职业时，来源地块必须有足够该职业人数。
- 未指定职业时，会按 `idle`、`builder`、`miner`、`lumberjack`、`farmer` 的顺序自动选择可迁职业。
- 目标地块必须有足够容量接收迁民。
- 来源地块迁出后必须仍有至少 1 名本族平民或 1 名本族士兵守住。
- 若来源地块有士兵守住，可以迁走最后一批平民。
- 迁移闲置人口可用于扩大领土、包围出生地、形成外圈扩张链。
- 把农民迁到平原、伐木工迁到森林、矿工迁到丘陵，有利于扩大对应资源产量。
- `fortify` 只能用于本族领土，执行效果相当于按 `workers=10` 训练士兵。
- `abandon` 只能用于本族领土，会清空该地块归属、房屋和本族士兵/人口。

## 八、视野与发现

- 每族可见范围是所有本族领土周围曼哈顿距离 2。
- 被淘汰种族没有视野。
- 扩张候选只列出可见、无主、可通行、邻接本族领土，且有安全迁民来源的格子。
- 边境敌方目标只列出可见、相邻、可通行、有敌方归属的格子。
- 种族发现来自被动视野；发现后才能用于外交和战争规划。

## 九、战争

- 军事动作：`muster`、`defend`、`attack`、`raid`、`retreat`、`move`。
- `move` 只能在相邻本族领土之间移动士兵。
- `attack` / `raid` 必须有 `origin` 和 `target`。
- 军事命令的 `origin` 必须是当前拥有且有士兵的本族地块。
- 军事目标必须与来源相邻，使用曼哈顿距离，不含斜角。
- `attack` / `raid` 目标必须是敌方领土，不能打无主地或自己。
- 目标被上帝保护时不能攻击。
- 如果目标是盟友，必须在同一计划中先提交 `war` 外交命令。
- 战斗发生后双方外交关系自动变为 `war`，双方互相加入已发现种族。
- 出兵数：`max(1, int(available * force_ratio))`，并且不超过来源地当前士兵数。
- `force_ratio` 必须大于 0 且小于等于 1。
- 森林和丘陵防守力乘以 `1.35`。
- 风暴目标会让进攻力乘以 `0.75`。
- 攻击力大于防守力时进攻成功。
- 进攻失败：进攻方士兵离开来源地并损失，防守方士兵减少 `attackers // 3`。
- `raid` 成功：幸存士兵返回来源地，目标人口受轻度损失，防守方按领土均摊比例损失资源，进攻方获得这些资源，不改变领土归属。
- `attack` 成功：目标地块改归进攻方，防守方该格人口和士兵清零，幸存进攻士兵进入目标地块，并夺取资源。
- 攻占不需要从攻击来源迁入闲置人口；占领先由幸存士兵守住。之后可用 `settle` 把平民迁入开发。
- 战斗事件会同时写给攻击方和防守方，因此双方首领都能在近期事件中看到。
- 每族有一个出生地。出生地被敌方占领时，该族立即淘汰。
- 淘汰后，该族人口和士兵归零，剩余资源转移给占领方，后续不再进行首领回合。
- `retreat` 目前可通过枚举，但执行层没有实际效果。

## 十、外交

- 外交动作：`alliance`、`trade`、`non_aggression`、`tribute`、`peace`、`war`。
- 不能对自己外交。
- 目标种族必须存在且已发现。
- 外交关系双向同步。
- `war` 变为战争。
- `peace` 变为中立。
- `trade` 变为贸易。
- `tribute` 变为纳贡。
- `non_aggression` 变为互不侵犯。
- 从 `neutral` 直接提 `alliance` 不会立刻同盟，只会先变成 `non_aggression`。
- 当前已经是 `non_aggression`、`trade` 或 `allied` 时，再提 `alliance` 才会变成 `allied`。
- 重复设置相同外交关系不会刷事件日志。
- 当前贸易/纳贡关系本身没有被动收益，只影响关系状态和首领决策倾向。

## 十一、首领 LLM

- 首领只通过游戏工具行动，不能直接修改世界。
- 战略首领唯一工具是 `submit_leader_turn`；`inspect` 工具已移除。
- `_build_leader_task` 会直接提供当前观察快照，包括资源、人口、士兵、职业、己方地块、扩张候选、迁民来源、边境目标、近期事件和已发现阵营可见摘要。
- 首领必须调用 `submit_leader_turn` 结束战略回合。
- 首领输出给玩家看的叙事字段必须使用简体中文。
- 神谕私聊会作为最高叙事优先级进入战略上下文，但订单仍必须符合当前规则。
- 首领战略上下文只保留最近 1 个战略回合。
- 长期记忆不再注入战略决策，不调用 LLM 压缩；当前只程序化保留神聊叙事和最近 3 条规则错误。
- `LLMLeaderController` 默认每次首领调用最多 8 个工具步。
- `GameEngine.retry_limit` 当前构造默认值为 `4`；含首次提交在内，最多会给同一首领多次修正机会。Web 入口未额外覆盖该值。
- 如果首领计划非法，规则错误会反馈给同一首领重试，并记录到规则错误记忆。
- 如果超过重试限制、缺少首领 controller、LLM 调用异常，世界会暂停。
- 被淘汰种族会跳过首领回合。

## 十二、首领提交协议

- `turn_intent` 必填。
- 可选字段：`population_orders`、`resource_orders`、`territory_orders`、`military_orders`、`diplomacy_orders`、`petitions`、`public_decree`、`strategy_summary`。
- `priority` 越小越先执行。
- 所有 `workers`、资源 `amount`、迁民 `amount` 不能为负；迁民 `amount` 必须为正。
- 资源消耗不能超过库存。
- 职业任务和训练不能超出可用闲置人口/人口预算。
- `claim` / `settle` 迁民量、工作分配和训练消耗会共同参与预算校验。
- 计划文字必须和实际订单一致：说要耕作、建房、进攻、扩张、祈求资源/天气，就必须提交对应订单；否则应避免相应措辞。
- 和平扩张应使用 `claim` / `settle` 语义，避免用攻击、战争、袭击等词。

## 十三、祈求

- 合法祈求类型：`resources`、`weather`、`protection`、`territory`。
- 祈求必须有 `reason`。
- `resources` 必须包含合法资源和正数数量，单次资源祈求上限 `250`。
- `weather` 必须包含可见坐标和合法天气。
- `protection` 必须包含可见坐标。
- `territory` 必须包含可见、无主、可通行、邻接本族领土的坐标。
- 同族同类未处理祈求会合并：
  - 资源祈求按 `faction + kind + resource` 合并。
  - 天气/保护/领土按 `faction + kind + x + y` 合并。
- 合并时资源数量取较大值，紧急度取较高值，理由和创建 tick 更新为最新。
- 拒绝祈求只改状态为 `rejected`。
- 批准祈求会执行对应上帝能力。

## 十四、上帝能力

- 上帝可以赐予资源。
- 上帝可以改变天气，也可以指定持续时间。
- 上帝可以保护或取消保护地块。
- 保护地块不能被攻击。
- 上帝可以批准首领的领土祈求；内部执行时仍要求目标可通行、邻接本族领土，并需要可迁平民来源。
- 上帝灾害支持：
  - `storm`：设置风暴。
  - `drought`：设置干旱。
  - `plague`：该格所有种族人口减少 `max(1, population // 4)`。
- Web UI 当前暴露：赐资源、改天气、批准/拒绝祈求、神谕私聊。
- 保护、灾害和内部直接划地属于系统能力，当前页面没有直接按钮。

## 十五、Web 与事件

- Web 坐标显示采用玩家视角 `(行, 列)`，内部规则仍使用 `x/y`。
- `/api/tick` 有异步锁；世界推进中再次请求会返回 `409`。
- `/api/god/chat` 也会占用同一把锁，避免战略推进和神聊同时改状态。
- Web 状态接口返回最近 80 条事件。
- 重要变化会写入事件日志，包括资源生产、人口增长、天气伤害、迁民、战斗、外交、祈求和上帝操作。
- 世界暂停原因保存在 `pause_reason`。
- `resume()` 可以恢复暂停，但当前 Web UI 没有恢复按钮。

## 十六、当前未完善或待扩展

- `trade` / `tribute` 会扣资源，但不会转移给对方。
- `reserve` 没有实际执行效果。
- `retreat` 没有实际执行效果。
- 科技、城市、文化、宗教、建筑细分、装备和更复杂经济链尚未实现。
- 战争已有攻击、占领、袭击和资源掠夺，但还没有士气、补给、远程、城防等系统。
- 首领策略依赖当前快照和规则校验；LLM 可能仍会提交非法计划，但会被规则引擎拦截并反馈重试。
