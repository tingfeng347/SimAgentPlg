# 完整规则说明

下面是按当前代码实现整理的完整规则，包括硬规则、软规则和一些“潜在但容易被忽略”的规则。

## 一、世界与地图规则

- 默认三族：`human` / `elf` / `orc`。
- 默认资源：`food` / `wood` / `stone`。
- 默认天气：`clear` / `rain` / `drought` / `storm`。
- 默认地形：`plain` / `forest` / `hill` / `water` / `mountain`。
- 默认地图是网格；默认世界生成是 `32x20`，当前 Web 示例启动的是 `12x8`。
- 地形随机生成：水约 8%，山约 8%，森林约 22%，丘陵约 17%，其余平原。
- 水和山不可通行，不可正常扩张。
- 每族初始有 1 格出生地领土。
- 出生地随机生成时会优先拉开距离：默认大图优先曼哈顿距离 8，小图空间不足时逐级降到 6/4/3/1。
- 每族出生地初始 15 人口、5 士兵、2 房屋，其中 10 人为农民、5 人闲置。
- 每族初始资源：`food=120`，`wood=80`，`stone=40`。
- 初始外交全是 `neutral`。

## 二、Tick 规则

- 每次 tick 会先让世界时间 `+1`。
- 如果世界已暂停，tick 不会继续推进。
- 每 tick 都会先处理天气衰减，再处理 NPC 被动结算。
- 每 5 tick 触发一次战略回合，也就是 LLM 首领回合。
- 普通 tick 不调用 LLM。
- 战略 tick 会并行调用三个首领思考。
- 战略 tick 中，所有首领提交合法计划后，再按种族排序执行 NPC 主动命令。
- 每 tick 结束会写入事件：`Tick N completed`。

## 三、天气规则

- `storm` 在 tick 数能被 3 整除时自动变回 `clear`。
- `rain` 在 tick 数能被 4 整除时自动变回 `clear`。
- `drought` 在 tick 数能被 5 整除时自动变回 `clear`。
- 注意：天气衰减发生在资源/人口结算之前，所以某些天气可能在本 tick 先消失，再结算效果。
- 雨天粮食倍率 `1.45`。
- 干旱粮食倍率 `0.2`。
- 风暴粮食倍率 `0.35`。
- 风暴还会让该格粮食产出额外减少 `2`。
- 风暴会让有人的格子损失 `1` 人口。
- 风暴在偶数 tick，如果该格有士兵，会损失 `1` 士兵。
- 干旱每 3 tick 造成一次人口损失，损失为 `max(1, population // 25)`。

## 四、资源增长规则

- 平原按人口产粮：`int(population * weather_multiplier / 14)`。
- 森林产木：`max(1, population // 16)`。
- 森林也产少量粮：`int(population * weather_multiplier / 34)`。
- 丘陵产石：`max(1, population // 20)`。
- 首领人口任务可以额外产资源：
  - `farm`：增加 `max(1, workers // 3)` 食物。
  - `gather_wood`：增加 `max(1, workers // 4)` 木材。
  - `mine_stone`：增加 `max(1, workers // 5)` 石料。
- 上帝可以直接赐予资源。
- 上帝批准 `resources` 祈求也会增加资源。

## 五、资源减少规则

- 每 tick 人口都会消耗食物：`max(1, total_population // 18)`。
- 食物不足时，剩余食物先清零，然后按缺口造成饥荒人口损失：`max(1, shortage // 2)`。
- 人口自然增长会消耗食物，增长多少就扣多少食物。
- 首领资源命令 `spend` / `trade` / `tribute` 会扣资源。
- 目前 `trade` / `tribute` 只扣本族资源，不会真的转移给对方，这是一个潜在未完善规则。
- `reserve` 会通过校验，但当前没有实际执行效果。

## 六、人口规则

- 人口不是 LLM，按规则自动增长/损失。
- 每 5 tick 检查一次自然增长。
- 食物低于 10 不会增长。
- 干旱和风暴格不增长。
- 地块有容量上限：
  - 平原：`90`
  - 森林：`65`
  - 丘陵：`45`
  - 其他：`20`
- 基础增长：`max(1, population // 30)`。
- 雨天额外 `+1` 增长。
- 增长不能超过地块容量，也不能超过当前可用食物。
- 训练士兵会把人口转成士兵，不是凭空生成士兵。

## 七、士兵规则

- `train` / `defend` / `muster` / `fortify` 会训练士兵。
- 训练数量大致为 `max(1, workers // 5)`，但最多不能超过该格人口的三分之一。
- 训练后人口减少，士兵增加。
- `fortify` 固定按 `workers=10` 训练。
- `muster` / `defend` 固定按 `workers=8` 训练。
- 风暴可杀士兵。
- 战斗会消耗进攻方士兵。

## 八、领土规则

- 首领只能声明可见坐标。
- 领土扩张目标必须在地图内。
- `claim` / `settle` 只能对无主或本族地块使用。
- 扩张目标必须可通行，不能是水或山。
- 扩张目标必须邻接本族已有领土。
- `fortify` / `abandon` 只能对本族领土使用。
- 定居无主地时，会从本族人口最多的地块迁移人口。
- 迁移规则：如果供给地人口大于 8，迁移 `min(8, population // 4)`。
- 如果没有足够人口，仍可占领无主地，但迁入人口可能是 `0`。
- 玩家不再直接划分领土。
- 领土神权只在批准 `territory` 祈求时通过内部能力触发。
- 内部上帝划地时，如果目标种族在该格没有人口，会自动放入 1 人口。

## 九、视野规则

- 每族可见范围是自己所有领土周围半径 2 的方形区域。
- 首领只能合法操作可见目标。
- `inspect_tiles` 查不可见格，只返回不可见，不暴露详情。
- `inspect_faction` 只统计对方可见领土、可见人口、可见士兵。
- 扩张候选只列出可见、无主、可通行、邻接本族领土的格。
- 边境敌方目标只列出可见、相邻、可通行、有敌方归属的格。

## 十、战争规则

- `attack` / `raid` 必须提供 `origin` 和 `target`。
- `move` 必须提供 `origin` 和 `target`，只能在相邻本族领土之间迁移士兵。
- 进攻起点必须是本族领土。
- 进攻起点必须有士兵。
- 目标必须相邻，使用曼哈顿距离，也就是上下左右，不含斜角。
- 目标必须是敌方领土，不能打无主地，不能打自己。
- 目标被上帝保护时不能攻击。
- 如果目标是盟友，必须在同一计划里先提交 `war` 外交命令，否则非法。
- 战斗发生后双方外交关系自动变为 `war`。
- 进攻士兵数：`max(1, int(available * force_ratio))`。
- `force_ratio` 必须大于 0 且小于等于 1。
- 目标是森林或丘陵时，防守力乘以 `1.35`。
- 目标有风暴时，进攻力乘以 `0.75`。
- 进攻力大于防守力则赢下战斗。
- 真正占领还需要进攻起点格有至少 1 个闲置人口、进攻起点格迁出后仍有人口、目标格容量足够。
- 占领迁入人口只能从发生战斗的进攻起点格迁到目标格，其他地块的闲置人口不能远程补占领。
- 占领后防守方该格士兵清零，进攻方幸存士兵进入目标格。
- 每个种族有一个出生地；出生地被其它种族占领时，该种族立即淘汰。
- 淘汰后该种族人口和士兵归零，剩余资源全部转移给占领方，后续不再进行首领回合。
- 占领敌方有人口的格子时，敌方人口损失 `max(1, population // 5)`。
- 进攻失败时，防守方士兵减少 `attackers // 3`。
- 隐含问题：`raid` 当前和 `attack` 逻辑一样，没有单独掠夺资源效果。

## 十一、外交规则

- 外交动作：`alliance` / `trade` / `non_aggression` / `tribute` / `peace` / `war`。
- 不能对自己外交。
- 目标种族必须存在。
- `war` 关系变为战争。
- `peace` 关系变为中立。
- `trade` 关系变为贸易。
- `tribute` 关系变为纳贡。
- `non_aggression` 关系变为互不侵犯。
- 从 `neutral` 直接提 `alliance` 不会立刻同盟，只会变成 `non_aggression`。
- 如果当前已经是 `non_aggression` / `trade` / `allied`，再提 `alliance` 才会变成 `allied`。
- 重复设置相同外交关系不会刷事件日志。
- 外交关系是双向同步的。
- 目前贸易/纳贡关系本身没有被动收益，只影响关系状态和 LLM 决策倾向。

## 十二、首领 LLM 规则

- 每个首领只能使用游戏工具，不能用 Bash 或文件系统。
- 首领工具只有：
  - `inspect_realm`
  - `inspect_tiles`
  - `inspect_faction`
  - `submit_leader_turn`
- 首领必须用 `submit_leader_turn` 结束回合。
- 每个首领最多工具步数默认 `8`。
- 战略回合中三个首领并行思考。
- 提交非法计划后，规则引擎把错误反馈给同一个首领。
- 默认最多重试 9 次，所以总共最多 10 次提交机会。
- 仍非法则世界暂停。
- 缺少某个首领 controller，世界暂停。
- LLM 调用异常，世界暂停。
- 首领 prompt 中的软规则：
  - 和平只是策略，不是默认结局。
  - 边境接触且有兵力时，应考虑战争、袭击、防御或威慑。
  - 不要重复提交已存在的外交关系。
  - 不允许向上帝祈求人口。
  - 只能祈求资源、天气、保护、无主领土。
  - Human 偏务实扩张。
  - Elf 偏防御与森林，但受威胁会先发制人。
  - Orc 偏激进劫掠和战争。

## 十三、首领命令协议规则

- `turn_intent` 必填。
- `population_orders` 可选，任务必须在枚举内。
- `resource_orders` 可选，资源和动作必须合法。
- `territory_orders` 可选，动作必须合法。
- `military_orders` 可选，动作必须合法。
- `diplomacy_orders` 可选，目标种族和提议必须合法。
- `petitions` 可选，类型和紧急度必须合法。
- 所有 `workers` 不能为负。
- 所有资源消耗不能为负。
- 总 `workers` 不能超过本族总人口。
- 优先级 `priority` 越小越先执行。
- 潜在规则：`build` / `attack` / `idle` 作为人口任务能通过枚举，但当前 NPC 执行层没有实际效果。
- 潜在规则：`retreat` 能通过军事枚举，但当前执行层没有实际效果。
- 侦察不再作为主动命令；种族发现来自已有领土半径内的被动视野。

## 十四、祈求规则

- 合法祈求类型只有：`resources` / `weather` / `protection` / `territory`。
- 祈求必须有 `reason`。
- `resources` 必须指定合法资源和正数数量。
- 单次资源祈求上限是 `250`。
- `weather` 必须指定可见坐标和合法天气。
- `protection` 必须指定可见坐标。
- `territory` 必须指定可见、无主、可通行、邻接本族领土的格子。
- 同族同类未处理祈求会合并：
  - 资源祈求按 `faction + kind + resource` 合并。
  - 天气/保护/领土按 `faction + kind + x + y` 合并。
- 资源数量取较大值。
- 紧急度取较高值。
- 理由和创建 tick 更新为最新。
- 拒绝祈求只改状态为 `rejected`。
- 批准祈求会执行对应上帝能力。

## 十五、上帝规则

- 上帝可以赐资源，资源和种族必须合法，数量必须大于 0。
- 上帝可以改变任意格天气。
- 玩家不再直接划分领土；领土神权只通过批准 `territory` 祈求间接触发。
- 内部上帝划地时不检查可见性、不检查邻接、不检查是否水山。
- 上帝可以保护或取消保护某格。
- 保护格不能被攻击。
- 上帝灾害支持：
  - `storm`：改为风暴。
  - `drought`：改为干旱。
  - `plague`：该格所有种族人口减少 `max(1, population // 4)`。
- Web UI 当前暴露了赐资源、改天气、批准/拒绝祈求；保护、灾害和直接划地属于内部系统能力，当前页面没有直接按钮。

## 十六、暂停与事件规则

- 世界暂停后，tick 会停止推进。
- 暂停原因会保存在 `pause_reason`。
- `resume()` 可以恢复，但当前 Web UI 没有恢复按钮。
- 所有重要变化都会写入事件日志。
- Web 只返回最近 80 条事件。
- Web tick 有异步锁；同一时间只能推进一个 tick 请求。
- 如果连续点击 tick，请求会被拒绝或排队行为由锁控制，目前锁住时返回 `409`。

## 十七、当前最重要的潜在未完善规则

- `trade` / `tribute` 会扣资源，但不会把资源给对方。
- `build` 没有建筑系统，所以暂无效果。
- `raid` 会掠夺资源但不会占领，尚未有更细的突袭后果系统。
- `retreat` 没有撤退实现。
- 科技、城市、建筑、文化、宗教、记忆还没有实际规则。
- 内部上帝划领土过于强，可以绕过全部地形/邻接/战争规则；玩家当前只能通过批准领土祈求间接触发。
- 普通 NPC 不会自主战略决策，只执行首领合法命令和被动生存规则。
- LLM 首领有策略倾向，但最终能否做成事由规则引擎和 NPC 执行层决定。
